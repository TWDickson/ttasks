Scope: architecture / SOLID / DRY / separation of concerns, TDD posture,
**date handling end-to-end**, the **repeat mechanism redesign**, and
**Obsidian publication readiness**. Verified against the live tree on
`feat/ui-polish-autopilot`.

Validation snapshot at audit time:

- `npm run test` — **1261 passing, 112 files**
- `npm run lint` — **10 errors** (`no-mixed-spaces-and-tabs`, all in
  `TaskGraph.svelte`)
- Working tree: `manifest.json` modified — **line-endings only** (CRLF↔LF,

## 1. Overall assessment

The codebase is in genuinely good shape — far better than the typical
community plugin. The strongest assets:

- **Pure-module discipline** — `src/query/`, `src/utils/`, `src/integration/`
  pure helpers, and the store decision modules (`completeTask`,
  `statusChanged`, `recurrence`, `reminderRules`) are Obsidian-free and
  enforced by `architectureBoundaries.test.ts`. This is the single best
  architectural decision in the repo; everything below builds on it.
- **Ports pattern** at the plugin boundary (`taskActionPorts`,
  `contextMenu` deps, `safeDelete` deps) keeps `main.ts` thin (642 lines,
  mostly wiring) and testable.
- **Every completion path routes through `decideCompletion`** — the
  recurrence idempotence guard lives in exactly one place, with the rationale
  documented at the call site.
- **Date model is documented and mostly honored** — `dateUtils.ts` states the
  hybrid local-date/UTC-arithmetic policy and the rest of the app largely
  follows it.
- 1261 tests with real behavioral coverage (not snapshot noise), a visual
  test rig, and store/component/contract test layers.

The weaknesses cluster in four places: (1) the old "big five" Svelte
components pre-date the coupling rule and violate it; (2) the Task field
schema is defined in ~4 places that must be updated in lockstep; (3) the
repeat mechanism is coarse **and has a real correctness bug** (monthly
end-of-month drift, RP-1); (4) publication scaffolding (README, LICENSE,
versions.json, CI, release workflow) doesn't exist yet.

## 2. Architecture / SOLID / DRY / separation of concerns

### AR-1 🔴 Component→plugin coupling rule is violated by all ten legacy components

CLAUDE.md: *"new components must not import `TTasksPlugin` or `TaskStore`
directly."* In practice every top-level component does:
`TaskBoard`, `TaskKanban`, `TaskList`, `TaskRow`, `TaskAgenda`, `TaskGraph`,
`TaskDetail`, `TaskDetailNotes`, `TaskDetailRelationships`,
`TaskArchiveView` all `import type TTasksPlugin from '../main'` and reach
through it. The rule only protects new code; the existing surface is
grandfathered and untestable via `@testing-library/svelte` without heavy
mocks.

**Plan** (incremental, one component per PR):

1. Define a `BoardContext` interface in `src/components/context.ts`: the
   narrow capability surface components actually use — `tasks: Readable<Task[]>`,
   `settings` snapshot + `settingsRevision`, `boardState`, and an actions
   object (`updateTask`, `setStatus`, `completeAndRecur`, `openDetail`,
   `showContextMenu`, `duplicate`, `archive`, …). Build it once in the three
   `views/*View.ts` leaves (which legitimately hold the plugin) and pass it
   down as a prop / Svelte context.
2. Migrate leaf-most components first (`TaskRow`, `TaskDetailNotes`), then
   containers (`TaskList`, `TaskKanban`, `TaskAgenda`), then the two big ones
   (`TaskDetail`, `TaskGraph` — coordinate with AR-2).
3. Enforce: add `src/components/**` to an ESLint `no-restricted-imports`
   rule banning `../main` and `../store/TaskStore`, so the rule stops being
   aspirational. Delete the CLAUDE.md caveat when done.

Acceptance: no `from '../main'` in `src/components/`; each migrated component
gains at least one render test that previously wasn't feasible.

### AR-2 🟡 `TaskGraph.svelte` is a 2,125-line god component

It contains pan/zoom state, pinch gestures, marquee hover, chain-highlight
pinning, diagnostics logging, lane rendering, edge rendering, and toolbar UI.
The only current lint failures (mixed tabs/spaces) live here too — a symptom
of too many hands in one file.

**Plan:**

1. Fix the 10 lint errors now (whitespace-only commit).
2. Extract in this order, each with tests where logic is pure:
   - `graphViewport.ts` — pan/zoom/pinch state machine (pure reducer +
     a thin Svelte action for pointer events).
   - `graphInteractions.ts` — chain-highlight pin/unpin, marquee hover
     (mostly pure already; move the state transitions out).
   - `GraphToolbar.svelte`, `GraphLane.svelte` subcomponents.
   - Diagnostics block (`console.info('[TTasks][GraphDiagnostics]'…`) behind
     the dev flag like `useTaskQuery` profiling.
3. Target: `TaskGraph.svelte` < 700 lines of composition + layout.

Note: GP7 (split Dependency/Timeline views, already in BACKLOG) will be much
cheaper after this extraction — sequence AR-2 before GP7.

### AR-3 🟡 Task field schema is defined in four places (DRY)

Adding a frontmatter field today requires touching, in lockstep:

1. `types.ts` (`Task` interface)
2. `TaskWriter.buildTaskFrontmatter` (hand-rolled YAML at create)
3. `TaskWriter.update` `writtenFields` array (persisted-field whitelist)
4. `TaskStore.fileToTask` (read/parse) — plus `schema/taskFields.ts` for UI

`workweek_only`/`holiday_dates` already show the strain (each has bespoke
validation in both TaskWriter and TaskStore, with a third copy of the ISO
regex in `taskGraphDates.ts`).

**Plan:** introduce a single field-descriptor table in
`src/schema/taskFields.ts` (it already half-exists for UI):

```ts
interface FieldDescriptor {
  key: keyof Task;
  kind: 'string' | 'date' | 'time' | 'number' | 'bool' | 'link' | 'links' | 'strings';
  persisted: boolean;          // appears in frontmatter
  writable: boolean;           // included in update() whitelist
  toYaml?: (v) => string;      // create-time serialization
  parse?: (raw) => value;      // load-time coercion
}
```

Derive `writtenFields`, `buildTaskFrontmatter`, and the `fileToTask`
coercions from it. This is the highest-leverage refactor in the repo: it
makes RP (repeat redesign) and any future field a one-row change, and the
`fmKey` mapping is what makes the MD-1 schema prefix (§3.5) a one-line-per-
field change instead of a codebase sweep.
TDD: port the existing TaskWriter/TaskStore tests over unchanged — they are
the safety net; the descriptor table gets its own round-trip test
(`task → yaml → parse → task`).

### AR-4 🟡 `TaskWriter` mixes four concerns (SRP)

684 lines covering: frontmatter CRUD, optimistic store patching, checklist
bidirectional sync (parent↔child), and recurrence spawning. The checklist
sync alone is ~200 lines with its own regex parser
(`syncParentChecklistFromChild` line 491 duplicates what
`utils/wikiLink.extractChecklistLink` parses).

**Plan:**

1. Extract `ChecklistSyncService` (`syncChildrenFromParentChecklist`,
   `syncParentChecklistFromChild`, `materializeChecklistChildrenFromBody`)
   — it depends on TaskWriter only via `create`/`update`/`completeAndRecur`,
   so inject those as a small port.
2. Reuse `extractChecklistLink` in `syncParentChecklistFromChild` instead of
   the inline regex (one checklist-line grammar, one parser).
3. Keep recurrence spawning in TaskWriter for now — it moves naturally in the
   RP redesign (§4).

### AR-5 🟢 Smaller DRY / correctness items

- **ISO date regex** `/^\d{4}-\d{2}-\d{2}$/` is duplicated in 8 files
  (settings/holidays, workingCalendar, protocol, CreateTaskModal, taskFields,
  taskDetailSaveController, TaskDetail.svelte, TaskWriter, TaskStore,
  taskGraphDates). Export `isIsoDateString(value): value is string` from
  `dateUtils` and sweep. (Also DT-6.)
- **`path.replace(/\.md$/, '')`** appears ~10 times in TaskWriter/main.ts
  despite `stripMdExt` existing in `pathUtils`. Sweep to `stripMdExt`/
  `ensureMdExt`.
- **First-status fallback** `settings.statuses[0] ?? 'Active'` is repeated
  3× in TaskWriter. Add `defaultStatus(settings)` next to
  `resolveCompletionStatus`.
- **`graphTimeline.ts` re-exports `dateUtils`** helpers and
  `taskGraphDates`/`hybridTimeline` import them through it. Import from
  `utils/dateUtils` directly; keep graphTimeline about timelines.
- **`isFileOpenInEditor` + `setTimeout(100)`** before `vault.process` in
  TaskWriter (2×) is a race-condition band-aid. `vault.process` is already
  atomic; either delete the sleep or document the specific editor-flush bug
  it works around (and file it upstream).
- **`as any`** in `TaskWriter.isFileOpenInEditor:421` — the only one left in
  src. Type it via `MarkdownView` (`leaf.view instanceof MarkdownView &&
  leaf.view.file?.path === …`).

### What I checked and did *not* flag

- Query engine design (filter tree, sort scopes, group caps) is clean and
  open for extension via field-keyed access — no changes needed beyond DT-2.
- Store decomposition (TaskStore 374 lines + Writer/Relationships/Migrations/
  Archive/Reminder services) is at a healthy altitude.
- Settings section-per-file layout, the `settings.ts` re-export shim, and the
  normalization-on-load flow are all sound. `onExternalSettingsChange`
  handles sync races correctly (re-entrancy guard + canonicalize).
- `main.ts` correctly avoids detaching leaves in `onunload`, uses
  `registerInterval`/`registerEvent` everywhere, and defers store load to
  `onLayoutReady` with a `resolved`-event re-sync. This is textbook.

## 3. Date handling (special attention)

The hybrid model documented at the top of `dateUtils.ts` (local calendar
dates for fields; UTC component math for arithmetic; instants only for
snooze/logbook timestamps) is the right design and is *almost* uniformly
applied. I traced every `new Date(`, `toISOString`, and inline date parse in
src. Instant-based usages that are **correct** and should stay:
`reminderSnooze` (snooze-until is a real instant), `ArchiveService` logbook
timestamps, `TaskStoreSyncQueue`/`TaskStore` perf timing, quiet-hours
`getHours()`.

Findings, worst first:

### DT-1 🔴 Agenda buckets and query results go stale at midnight

`query/engine.ts` calls `localDateString()` internally (`resolveDate`,
`classifyAgendaBucket`, `classifyLogbookBucket`, `within_days`). Nothing
re-runs the query at midnight: `useTaskQuery` recomputes only when tasks or
the spec change, and only `TaskRow`/`TaskKanban` subscribe to the `today`
store. A board left open overnight shows yesterday's Overdue/Today buckets,
while the rows' relative badges (which do subscribe) update — a visible
inconsistency.

**Plan:**

1. Make the engine pure: `applyQuery(tasks, query, ctx: { today: string })`;
   thread `today` through `resolveDate`/bucket classifiers. All engine tests
   become deterministic (no more implicit wall-clock).
2. In `useTaskQuery`, derive the result from `[tasks, query, today]` using
   the existing `todayStore` — midnight flip re-queries everything for free.
3. Sweep the remaining "today" surfaces onto the store or an injected
   parameter: `TaskGraph`/`hybridTimeline`'s today-marker
   (`startOfToday()` captured at mount goes stale in a long-lived leaf),
   `TaskBoard`, `TaskDetail`, `statusSummary` (status bar recomputes on task
   changes only — an overdue count can be wrong after midnight until a task
   changes).

TDD: engine tests pass explicit `today`; add one `todayStore`-driven test
that flips the date and asserts re-bucketing.

### DT-2 🔴 `due_time` is stored but semantically dead

`Task.due_time` ('HH:MM') is persisted, whitelisted in `writtenFields`,
sortable, and offered in the query editor — but **nothing** consumes it:
overdue logic (`taskDateMeta.isTaskOverdue`), agenda bucketing, and all four
reminder rules are date-only. A task due today 09:00 shows no reminder at
10:00 and is not "overdue" until tomorrow.

**Decision needed, then implement:**

- **Option A (recommended): make it real.** Reminders gain a
  `due-time-passed` rule (task due today, `due_time` < current HH:MM,
  not complete). The 5-minute poll loop already exists; the rule is pure and
  slots into `evaluateReminders(task, today, nowHHMM, …)`. Overdue *styling*
  stays date-based (a task isn't "overdue-red" at 09:01) — document that.
- **Option B: declare it display/sort-only** in types.ts and the (future)
  README, and remove it from any reminder-adjacent expectations.

Either way, `dateUtils` grows the one missing primitive:
`localTimeString(now): 'HH:MM'` — keep `new Date()` out of the rules.

### DT-3 🔴 Monthly/yearly recurrence drift (see RP-1)

`advanceDate('2026-01-31','monthly')` → `2026-02-28` → **`2026-03-28`** →
`04-28`… The clamp is applied to the *previous occurrence* rather than the
*rule's anchor day*, so an end-of-month schedule permanently drifts to the
shortest month's day. Verified by execution. Fix lands with the repeat
redesign (§4) — the new model computes each occurrence from the rule, not
from the last clamped date. If the redesign is deferred, patch `advanceDate`
to carry the anchor day (add an `anchorDay` param defaulted from the date).

### DT-4 🟡 `recurrence.ts` contradicts the dateUtils contract

The doc comment says *"Parses at T12:00:00 local time"* but the code uses
`'T12:00:00Z'` + UTC accessors. The **behavior is correct** (pure calendar
math in a DST-free frame) but the comment is wrong, and the module violates
`parseIsoDate`'s "single canonical parser" claim with its own parse/format.
Fold `advanceDate` onto `dateUtils` primitives during the RP redesign; fix
the comment immediately. The monthly/yearly branches also duplicate the
days-in-month clamp — extract once.

### DT-5 🟡 "This week" is a rolling 7 days, not a calendar week

`classifyAgendaBucket`: `this-week` = due ≤ today+7, `next-week` = ≤ today+14.
On a Friday, "This Week" contains next Thursday. TickTick/Things use calendar
weeks. Decide intentionally: either rename the buckets ("Next 7 days" /
"Following week") or implement calendar-week bucketing with a
week-starts-on setting. Cheap either way because the bucket keys/labels are
already centralized in `agendaBuckets.ts` — but do it before publication,
because bucket names are user-facing semantics you'll be stuck with.

### DT-6 🟢 Consolidation and enforcement

- Add `isIsoDateString` to dateUtils; sweep the 8 duplicate regexes (AR-5).
- `taskDateMeta.formatHumanDate` does manual string slicing — fine, but move
  it into dateUtils next to `MONTH_ABBR` so all human-formatting lives
  together; `relativeDueDate`'s fall-off (>7 days shows raw ISO) is a UX
  choice worth revisiting alongside DT-5.
- **Enforce the boundary**: extend `architectureBoundaries.test.ts` (or an
  ESLint `no-restricted-syntax` rule) to ban `new Date(` outside
  `utils/dateUtils.ts`, `todayStore.ts`, and an explicit allowlist of
  instant-based files (reminderSnooze, ArchiveService, sync-queue timing,
  ReminderService quiet-hours). Every DT finding above was a drift from the
  documented model; a guard stops the next one.

### Verified-sound (no action)

- `parseIsoDate` correctly rejects malformed and impossible dates (ISO
  parsing NaNs out-of-range components) and normalizes to local midnight.
- `daysBetweenLocal`/`addDaysLocal` DST reasoning is correct.
- `todayStore` single-timer design is right; it just needs more subscribers
  (DT-1).
- Working-calendar math (`addCalendarDays`, holiday/recurring-holiday sets,
  per-area workweek with legacy fallback) is pure, well-commented, and
  test-covered. The Kahn-propagation date resolver with completed-work
  anchoring is genuinely good.
- Filename/emoji parsers in `src/integration/` treat dates as opaque strings
  matched by pattern — consistent with the model.

## 3.5 Frontmatter & property-pool hygiene (MD)

Added after review discussion. **Decisions settled with Taylor 2026-07-12:**
all plugin keys get a `ttask_` prefix, frontmatter is written sparsely,
`blocks` stops being persisted, and the vault is converted by a one-shot
script — **no legacy read shims in the plugin code**. The dev-phase
migration commands get deleted at the same time.

### How Obsidian metadata actually behaves

- **Property names are vault-global.** Every frontmatter key ever seen gets
  an entry in `.obsidian/types.json` and appears in the "Add property"
  autocomplete and All Properties view for every note. No namespacing, no
  per-folder schemas.
- **Types are per-name, vault-wide** (Text/List/Number/Checkbox/Date/…).
  Two systems using the same name with different shapes produce
  type-mismatch warnings in the Properties panel.
- **No public API** — `metadataTypeManager` is private; a plugin cannot
  register, type, or scope its properties. Write discipline is the only
  lever.

Evidence from the live vault's `types.json`: ~15 TTasks keys sit in the
global pool; `parent_task` and `source` are mis-typed as **`aliases`**
(a core type with special behavior); dead `category`/`task_type` entries
survive from pre-Phase-6; generic names (`type`, `name`, `status`,
`source`, `created`, `completed`) are claimed vault-wide. In-file, a
typical task carries ~24 frontmatter lines (`completed: null`,
`blocked_reason: ""`, `blocks: []`, …) where ~8 carry information.

### MD-1 🔴 Prefix the schema: `ttask_*`

Every plugin-owned key is renamed with the `ttask_` prefix (matching the
`ttask` cssclass): `ttask_type`, `ttask_name`, `ttask_area`,
`ttask_status`, `ttask_priority`, `ttask_labels`, `ttask_parent_task`,
`ttask_depends_on`, `ttask_blocked_reason`, `ttask_assigned_to`,
`ttask_source`, `ttask_start_date`, `ttask_due_date`, `ttask_due_time`,
`ttask_estimated_days`, `ttask_workweek_only`, `ttask_holiday_dates`,
`ttask_created`, `ttask_completed`, `ttask_status_changed`, plus the
`ttask_repeat_*` family (§4). `cssclasses` stays — it is a core Obsidian
key, and the whole point is that we *don't* own it. `blocks`,
`recurrence`, and `recurrence_type` disappear entirely (MD-3, §4).

Implementation notes:

- **The in-memory `Task` model keeps its current key names**
  (`task.due_date`, query specs, saved Smart Lists in `data.json`, protocol
  params — all untouched). Only the serialization boundary prefixes, via
  the AR-3 descriptor table's `fmKey` column. This is why AR-3 lands first.
- After the descriptor table, sweep the remaining raw `fm.` accesses that
  bypass it: `TaskWriter`'s `processFrontMatter` callbacks
  (`fm.depends_on`, `fm.blocks`, `fm.status`), `TaskMigrations`,
  `completionSync`, and the graph's `holiday_dates` parse.
- Published users start on the prefixed schema from day one — the generic-
  name collision problem (someone's web-clipper `source`, someone's
  Checkbox-typed `completed`) never happens.

### MD-2 🔴 Sparse writes

Only write keys that hold a value at creation; on update, **delete** a key
when its value clears (`delete fm[key]`) instead of writing `null`.
Verified: `fileToTask` already defaults every missing key, so sparse files
load correctly with no read-side change. Descriptor table gets an
`omitWhenEmpty` flag; the current `?? null` idiom in `TaskWriter.update`
goes away. A non-repeating, unblocked task drops from ~24 frontmatter
lines to ~8.

### MD-3 🟡 Stop persisting `blocks`

`blocks` is a pure reverse index of `depends_on`, kept consistent by a
whole subsystem (`TaskRelationships`, `addToBlocks` on every create/link,
the `sync-blocks` repair command). Compute it in memory at store-load
instead; UI is unchanged. Graceful degradation survives because Obsidian's
native backlinks pane already shows exactly this information (the
`depends_on` wiki-links point at the blocked task). Deletes the largest
source of multi-file write fan-out in the plugin.

### MD-4 🟡 One-shot vault migration + dev-command pruning

A standalone script (`Scripts/migrate-prefixed-schema.mjs`, run once with
Obsidian closed against `~/Obsidian/Taylor`):

1. For every task file (tasks folder + archive folder): rename keys per the
   MD-1 map, drop `blocks`, map legacy `recurrence`/`recurrence_type` to
   `ttask_repeat_*` (§4), strip null/empty keys (MD-2).
2. Verify by launching Obsidian and checking the board + a spot-check task.

Because the vault is migrated externally, the plugin ships **zero legacy
code** — and the dev-phase commands (`migrate-phase6-data-model`,
`migrate-status-changed`, `migrate-css-classes`, `archive-migrate-completed`
stays) plus their `TaskMigrations` implementations are deleted. Published
users never had the old schema; Taylor's vault no longer does either.

### MD-5 🟢 Registry cleanup (manual, 5 minutes)

After migration, hand-edit the vault's `types.json` (Obsidian closed):
remove the old generic entries (`type`, `name`, `status`, `priority`,
`parent_task`, `depends_on`, `blocks`, `blocked_reason`, `assigned_to`,
`source`, `start_date`, `due_date`, `created`, `completed`,
`estimated_days`, `category`, `task_type`) so they stop appearing in the
suggestion pool, and let the `ttask_*` names get typed on first use
(set the date fields to Date type for nicer Bases behavior). Document the
recommended property types in the README for other users.

## 4. Repeat mechanism — redesign for granularity

**Decisions settled with Taylor 2026-07-12** (supersedes the human-DSL
proposal from the first draft): storage is **flat, prefixed frontmatter
keys** (`ttask_repeat_*`), not a human-language string and not a nested
object. Rationale: the UX survey (§4.3) showed a builder must be the
primary entry path anyway, which makes a DSL a lossy round-trip layer with
its own parser to maintain; nested YAML objects are second-class in
Obsidian (Properties panel renders them as an uneditable blob, Bases can't
reach into them). Flat scalar/list keys are native everywhere — Properties,
Dataview, Bases (N7). No legacy compatibility: the vault is converted by
the MD-4 script.

### Current state and defects

`recurrence: daily|weekly|biweekly|monthly|yearly` +
`recurrence_type: fixed|from_completion`; completion spawns a new file via
`completeAndRecur`, deduped by `decideCompletion`'s guard.

- **RP-1 bug (verified by execution):** end-of-month drift — Jan 31
  monthly → Feb 28 → **Mar 28 forever**, because each occurrence is
  computed from the previous *clamped* date instead of the rule's anchor.
- **RP-2 expressiveness:** no "every N", weekday sets, nth-weekday,
  weekday classes, end conditions, or working-day awareness.
- **RP-3 fragile identity:** the spawn guard matches on task `name`;
  renaming a recurring task with an open instance breaks dedupe.

### Storage: flat `ttask_repeat_*` keys

```yaml
# every 2 weeks on Mon/Wed
ttask_repeat_every: 2
ttask_repeat_unit: week
ttask_repeat_weekdays: [mon, wed]

# 3 days after completion
ttask_repeat_every: 3
ttask_repeat_unit: day
ttask_repeat_basis: completion

# first working day of the month, 12 times
ttask_repeat_every: 1
ttask_repeat_unit: month
ttask_repeat_nth: 1
ttask_repeat_nth_target: working-day
ttask_repeat_count: 12
```

| Key | Type | Applies to |
| --- | --- | --- |
| `ttask_repeat_every` | number ≥ 1 | all — presence means "repeats" |
| `ttask_repeat_unit` | `day\|week\|month\|year` | all |
| `ttask_repeat_weekdays` | list of `mon…sun` | week |
| `ttask_repeat_monthday` | number or `last` | month; year (with `_month`) |
| `ttask_repeat_month` | 1–12 | year |
| `ttask_repeat_nth` | 1–5 or -1 (last) | month |
| `ttask_repeat_nth_target` | `mon…sun`, `day`, `weekday`, `weekend-day`, `working-day` | month |
| `ttask_repeat_basis` | `completion` (omit = schedule) | all |
| `ttask_repeat_roll` | `next\|previous` (omit = keep) | all |
| `ttask_repeat_until` | YYYY-MM-DD | end condition |
| `ttask_repeat_count` | number (remaining spawns) | end condition |
| `ttask_repeat_of` | wiki-link to the previous instance | spawned instances (RP-3) |

Yearly reuses `_monthday` + `_month` rather than inventing a third date
shape. Typical rules touch 2–4 keys; non-repeating tasks have zero
(sparse-write discipline, MD-2). Every value is a scalar or string list —
first-class in Properties, Dataview, and Bases.

### Typed model and modules (`src/repeat/`, pure)

```ts
export interface RepeatRule {
  every: number;                          // ≥ 1
  unit: 'day' | 'week' | 'month' | 'year';
  weekdays?: Weekday[];                   // unit=week
  monthday?: number | 'last';             // unit=month|year; mutually exclusive with nth
  month?: number;                         // unit=year
  nth?: { n: 1 | 2 | 3 | 4 | 5 | -1; target: NthTarget }; // unit=month
  basis?: 'schedule' | 'completion';      // default 'schedule'
  roll?: 'next' | 'previous';             // working-day roll; default keep
  until?: string;                         // YYYY-MM-DD
  count?: number;                         // remaining occurrences
}
export type NthTarget = Weekday | 'day' | 'weekday' | 'weekend-day' | 'working-day';
```

- `repeat/normalize.ts` — `normalizeRepeat(fm: Record<string, unknown>):
  RepeatRule | null`. The read boundary: gathers `ttask_repeat_*` keys,
  coerces types, enforces invariants (unit=month needs exactly one of
  monthday/nth; weekdays only with unit=week; …). An inconsistent
  hand-edited combo normalizes to `null` **with a warning badge on the
  task** — never a silent guess. There is deliberately no natural-language
  parser; this module replaces it and is an order of magnitude simpler.
- `repeat/describe.ts` — `describeRepeat(rule): string` for UI labels
  ("Every 2 weeks on Mon, Wed"). Display-only; nothing parses it back.
- `repeat/next.ts` — `nextOccurrence(rule, anchor: string, after: string,
  calendar?: WorkingCalendar): string | null`. Semantics:
  - **Anchor-based, never last-occurrence-based**: monthly `day 31`
    re-derives from the rule each month — Jan 31 → Feb 28 → **Mar 31**
    (RP-1 fixed by construction).
  - `basis: 'completion'` anchors on the completion date.
  - `working-day`/`weekend-day` targets and the `roll` option consult the
    existing `WorkingCalendar` (already Obsidian-free in `taskGraphDates`;
    export it from a shared location). Holiday-aware recurrence — "first
    working day of the month" skipping Jan 1 — is a genuine differentiator
    no mainstream task app offers.
  - Returns `null` when `until`/`count` is exhausted → the completion path
    simply doesn't spawn.

Settled edge semantics (encode these in the test table):

- "Last weekend" means **last weekend day** (Things/Apple semantics —
  usually the final Sunday); the builder preview disambiguates for users.
- **Nth that doesn't exist** ("5th Tuesday" in a 4-Tuesday month): **skip
  the month** (Things behavior) — clamping reintroduces drift-shaped
  surprises.
- Leap-day yearly (Feb 29): clamp to Feb 28 in non-leap years.
- v1 excludes multiple month-days ("day 1 and 15"); `monthday: number[]`
  extends cleanly later, but it complicates the builder for a rare case.

### Integration

1. `decideCompletion`/`completeAndRecur` call `nextOccurrence`; start-date
   offset preservation becomes `nextStart = addDaysLocal(nextDue,
   -(daysBetweenLocal(start, due)))` — one code path.
2. **RP-3:** spawned instances carry `ttask_repeat_of` (aliased wiki-link
   to the completed instance); the already-spawned guard matches on that
   link, not `name`. No name-heuristic fallback — the MD-4 script converts
   the vault, so legacy instances don't exist.
3. `count` decrement: spawned instance is written with `count - 1`.
4. `recurrence.ts` is deleted; its tests are ported to `repeat/next` before
   removal.

### UX: three layers over one model

All three layers read/write the same `RepeatRule`; nothing parses text.

1. **Contextual presets** (tap the Repeat field → popover on desktop,
   bottom sheet on mobile), derived from the task's due date, Google
   Calendar-style: "Every week on Fri", "Every month on the 2nd Friday",
   "Every month on day 13", "Every year on Jun 13", "Every weekday".
2. **Custom builder**:

   ```text
   Every [ 3 ▲▼] [days | weeks | months | years]

   (weeks)   [M][T][W][T][F][S][S]              ← multi-select chips
   (months)  ◉ On day [13 ▾]
             ○ On the [first ▾] [weekday ▾]     ← Mon…Sun | day | weekday |
             ○ On the last day                     weekend day | working day
   (years)   On [Jun ▾] [13 ▾]

   Repeat from   ◉ Schedule   ○ Completion
   If it lands on a non-working day   [Keep | Next working day | Previous]
   Ends   ◉ Never   ○ On [date]   ○ After [10] times
   ──────────────────────────────────────────
   Next: Jul 15 · Jul 29 · Aug 12               ← nextOccurrence ×3
   ```

   The **next-3-occurrences preview** is the highest-value element — it is
   how users verify "first working day of the month" does what they meant,
   and it is nearly free.
3. **Raw editing** happens in the file itself: the `ttask_repeat_*` keys
   are plain YAML, and an invalid combination surfaces as a warning badge
   via `normalizeRepeat` rather than being silently dropped.

Mobile: bottom sheet per existing modal conventions; golden-path test on a
narrow viewport before closing (per CLAUDE.md). Add the builder popover to
the rig workshop list for dark/light shots before implementation.

### TDD plan (write these first)

- `normalizeRepeat`: table-driven valid/invalid/inconsistent-combination
  cases; unknown keys dropped; every invariant has a rejecting test.
- `nextOccurrence`: table tests per unit; named edge cases — Jan 31
  monthly ×12 (no drift), leap-day yearly, `last` monthday, missing nth
  (skip month), weekday sets spanning week boundaries, DST spring/fall
  weeks, `until` landing exactly on an occurrence, `count: 1`,
  working-day roll across a holiday run, `basis: completion` with late and
  early completions.
- Round-trip: `RepeatRule → ttask_repeat_* keys → normalizeRepeat ≡ id`
  over generated rules (this replaces the old DSL round-trip law).
- Completion integration: port `completeTask.test.ts`/`recurrence.test.ts`
  to the new engine before deleting the old one; new tests for
  `ttask_repeat_of` dedupe and rename-safety.

### Sequencing

AR-3 descriptor table → MD-4 vault migration script (prefix + sparse +
blocks + legacy-recurrence mapping, run once) → repeat engine (pure
modules + integration) → builder UI. The engine PR ships behind the
existing two dropdowns if needed, but with no legacy formats in play the
cleanest cut is engine + UI in consecutive PRs.

## 6. Obsidian guidelines + publication prep

Measured against the official plugin guidelines and the community-plugin
review bot's known flags.

### PB-1 🔴 Missing release scaffolding

None of these exist yet:

- **README.md** — required by submission. Must cover: what it does,
  screenshots (the rig can generate the desktop/mobile × dark/light matrix),
  the frontmatter data model (a selling point — "your tasks are plain
  markdown"), the date-handling policy (local calendar dates; document the
  timezone-travel caveat), and settings overview.
- **LICENSE** — required (MIT is the ecosystem default).
- **versions.json** — required for release (`{"0.1.0": "1.7.2"}`).
- **Release workflow** — GH Action on tag: build, attach `main.js`,
  `manifest.json`, `styles.css` to a GitHub release whose tag exactly equals
  `manifest.json` version (no `v` prefix — the submission checker enforces
  this). Add a `version-bump.mjs` that syncs package.json → manifest.json →
  versions.json (the standard sample-plugin script).
- Then: PR to `obsidianmd/obsidian-releases` adding the plugin to
  `community-plugins.json`.

### PB-2 🔴 Review-bot flags in the code

These are the things the automated + human review will bounce:

- **`innerHTML` assignments** (5 sites): `QueryEditorModal` sets
  `btn.innerHTML = '✕'` (3×) — also violates the repo's own Lucide-only
  convention; use `setIcon(btn, 'x')`. `CreateTaskModal:557` and
  `TaskDetailNotes.svelte:40` clear with `innerHTML = ''` — use `el.empty()`.
- **`app.workspace.activeLeaf`** (`TaskBoardView.ts:58`) — deprecated;
  replace the "is this leaf active" check with
  `this.app.workspace.getActiveViewOfType(TaskBoardView) === this` or
  compare against `workspace.getMostRecentLeaf()`.
- **`vault.modify`** (`completionSync.ts:96`, `vaultSafe.ts:46`) — guidelines
  require `Vault.process` for read-modify-write (atomic against sync/
  concurrent edits). `completionSync` reads earlier via `cachedRead` then
  modifies — a real clobber window. Move the line-edit into a
  `vault.process` callback.
- **Console noise in production**: `plugin.log` → `console.log`
  unconditionally, plus `TaskStore.ts:278` per-file skip logs. Gate behind
  the same `process.env.NODE_ENV` flag used by `useTaskQuery` (reviewers
  flag chatty consoles). `console.error` for real failures is fine and
  should stay.
- **`localStorage` via hand-rolled vault namespacing**
  (`reminderStorage`/`vaultSafe`): Obsidian provides
  `app.loadLocalStorage()`/`app.saveLocalStorage()` which namespace per
  vault (incl. mobile). Swap the backing calls inside `vaultSafe` — call
  sites unchanged. Per-device semantics for fired reminders are correct and
  documented; keep them.
- **Casing / headings**: UI text must be sentence case — sweep
  `'Edit Smart List:'`, kanban/settings headings, command names (current
  commands are compliant). `managedListSettingsSection.ts:46` uses
  `createEl('h3')` in the settings tab — convert to
  `new Setting(el).setName(...).setHeading()` like every other section.
  Modals using `createEl('h2')` should use `this.titleEl.setText(...)`
  (keeps the native modal header semantics).

### PB-3 🟡 Manifest polish

- `description` contains "plugin for Obsidian" — the checker flags both
  words as redundant. Suggested: *"Task management with kanban, dependency
  tracking, agenda, and a dependency graph — stored as plain markdown
  frontmatter."*
- Verify `minAppVersion: 1.7.2` against the newest API actually used
  (`workspace.ensureSideLeaf`, `vault.getFileByPath`, `setTooltip` are all
  ≥1.4–1.5; 1.7.2 is safe, but re-check when adopting anything new).
- Optional: `fundingUrl`.
- The line-ending churn (top of this doc): commit LF + `.gitattributes`.

### PB-4 🟡 Svelte CSS is JS-injected, contradicting the design-system rule

`esbuild-svelte` runs with `compilerOptions: { css: 'injected' }` — every
component's scoped styles are injected as `<style>` elements at runtime,
while CLAUDE.md's design system says "No JS-injected `<style>` elements."
The rule was written for hand-injected styles, but the effect is the same:
component CSS bypasses `styles.css`, can't be overridden by theme snippets
predictably, and briefly FOUCs on view open. **Plan:** switch to
`css: 'external'` and have `esbuild.config.mjs` concatenate the emitted CSS
onto `styles.css` at build (sample-plugin ecosystem has established patterns
for this). Verify with the rig (`npm run rig:shots`) that nothing regresses.
Not a submission blocker, but do it before styles.css becomes a public API
for theme authors.

### PB-5 🟢 Things that are already publication-clean

- No network calls, no telemetry, no Node/Electron imports in src (esbuild
  externals list is belt-and-braces), `isDesktopOnly: false` matches the
  mobile support.
- `fileManager.processFrontMatter` for all frontmatter mutation; raw YAML
  only at file creation (documented convention).
- No leaf detaching in `onunload`; intervals/events all registered for
  cleanup; ribbon/command naming compliant; `normalizePath` used at vault
  boundaries.
- Command `seed-graph-test-data` is correctly dev-gated out of production.
- `data.json`, `main.js` correctly gitignored.
